### YUV_Cleaner.bat 原有逻辑问题分析

1. **清理时机依赖间接信号**：bat 通过轮询检测**下一个** chunk 的 `_bl.yuv` 是否存在来决定删除当前 chunk 的 YUV 文件。虽然在顺序执行中功能正确，但这是间接判断——实际安全点应是当前 chunk 的 Vesmux 完成后（即所有 YUV 消费者都已结束）。
2. **轮询机制脆弱**：300 秒固定轮询间隔需要用户手动调整，且 bat 作为独立进程运行，与 Python 工作流缺乏同步，存在竞态风险。
3. **倒数第二个 chunk 的计算逻辑**：bat 中 `calculateFinalNumber` 循环在超过 `extraNumber` 后减去了 `intervalLength*2`（应减 `intervalLength`），导致 `finalNumber` 实际指向倒数第三个编号而非倒数第二个——不过这取决于对"倒数第二"的定义和实际 chunk 边界。

### 集成方案的改进

| 方面          | bat（原）                                   | 集成后（新）                                                 |
| ------------- | ------------------------------------------- | ------------------------------------------------------------ |
| 清理时机      | 轮询检测下一 chunk 的文件存在               | 在每个 chunk 的 Vesmux 完成后**立即**清理，100% 安全         |
| 覆盖范围      | 所有 chunk（包括最后一个 chunk 需特殊处理） | 所有 chunk 统一处理，无需特殊逻辑                            |
| 清理目标      | `_bl.yuv`, `_bl_decoded.yuv`, `_el.yuv`     | 相同（这三个是最大的中间文件）                               |
| `--keep-temp` | 不支持                                      | 尊重该标志，设为 true 时跳过清理                             |
| `--dry-run`   | 不支持                                      | 通过 `Print_clean_chunk_yuv` 打印将被清理的文件              |
| 文件追踪      | 无                                          | 已清理的文件从 `tracked_files` 中移除，避免 `clean_temp()` 重复操作 |

### 具体改动（4 处）

1. **`FileManager.remove_from_tracking()`** — 新增方法，将已清理文件从追踪列表中移除
2. **`Clean_chunk_yuv`** — 新增类，在 Vesmux 后删除该 chunk 的 3 个 YUV 文件
3. **`Print_clean_chunk_yuv`** — 新增类，dry-run 模式下打印即将清理的文件
4. **`Workflow.get_run_dry_steps()` / `get_normal_run_steps()`** — 在每个 chunk 的 Vesmux 后插入清理步骤
